# GitHub-Discussions-README

GitHub Discussions ist der Raum für Gespräche, Wissen, Ideen und Community-Austausch.

Es ist **nicht** das Gleiche wie Issues und **nicht** das Gleiche wie Pull Requests.

## Was ist GitHub Discussions?

Discussions sind der richtige Ort für Themen, die noch **nicht** direkt eine konkrete Code-Änderung sind.

Zum Beispiel:

- Fragen
- Ideen
- Architekturgedanken
- Ankündigungen
- Feedback
- Community-Austausch

---

## Wann man Discussions nutzen sollte

Nutze Discussions für:

- „Wie sollten wir X angehen?“
- „Welche Richtung ist sinnvoll?“
- „Hat jemand Erfahrung mit Y?“
- „Hier ist ein Vorschlag / eine Idee“
- „Das ist neu im Projekt“

Kurz:

> Discussions sind für Austausch, Klärung und Wissensfluss.

---

## Wann man Discussions nicht nutzen sollte

Nutze **stattdessen Issues**, wenn:

- ein klarer Bug vorliegt
- eine konkrete Aufgabe umgesetzt werden soll
- Akzeptanzkriterien gebraucht werden

Nutze **stattdessen Pull Requests**, wenn:

- Code bereits geändert wurde
- Review nötig ist
- Merge bevorsteht

---

## Gute Kategorien für Discussions

Empfohlene Kategorien:

- **Announcements** – wichtige Neuigkeiten
- **Ideas** – neue Vorschläge
- **Q&A** – Fragen und Antworten
- **General** – allgemeiner Austausch
- **Show & Tell** – Dinge zeigen, die schon gebaut wurden
- **Decisions** – dokumentierte Richtungsentscheidungen

---

## Wie OpenSIN Discussions nutzen sollte

Für OpenSIN können Discussions besonders stark sein für:

- frühe Produktideen
- Tool- und Workflow-Vorschläge
- Architekturabstimmung vor Umsetzung
- Wissensaustausch im Team
- Community-nahe Kommunikation

Ein sinnvoller Flow ist:

1. Idee oder Frage startet in **Discussion**
2. daraus wird bei Bedarf ein **Issue**
3. Umsetzung läuft über **Branch + Pull Request**
4. Ergebnis kann wieder in der Discussion verlinkt werden

So gehen Gedanken nicht verloren und Umsetzung wird trotzdem strukturiert.

---

## Was eine gute Discussion ausmacht

Eine gute Discussion enthält:

- klares Thema
- verständlichen Kontext
- konkrete Frage oder Zielrichtung
- genügend Informationen für sinnvolle Antworten

### Gutes Beispiel

```text
Titel: Sollten wir für interne Dokumentation stärker auf GitHub Discussions setzen?

Kontext:
- aktuell viel Wissen in Chat-Verläufen
- wenig Wiederverwendbarkeit

Frage:
- Welche Themen sollten wir in Discussions statt in Issues pflegen?
```

---

## Moderationsregeln

Discussions funktionieren nur, wenn sie nicht verwildern.

Deshalb:

- Titel klar formulieren
- Themen nicht mischen
- Dubletten vermeiden
- Antworten konkret halten
- bei Umsetzungsbedarf in ein Issue überführen

---

## Gute Teamregel

Wenn ein Thema noch **offen**, **diskursiv** oder **erkundend** ist, beginnt es oft besser in Discussions als in einem Issue.

Wenn daraus jedoch echte Arbeit entsteht, sollte daraus ein Issue werden.

---

## Anti-Patterns

Vermeidet:

- Bugreports als Discussion statt Issue
- PR-Debatten ohne verlinkte Umsetzung
- endlose Diskussion ohne Entscheidung oder Übergabe
- unklare Titel wie „Frage“ oder „Idee“

---

## Kurzregel

> Discussions sind der Raum für Denken, Fragen, Einordnung und Richtung. Issues sind der Raum für konkrete Arbeit.

---

## Qualitäts-Check

- [ ] Die Discussion hat eine klare Kategorie
- [ ] Der Titel ist konkret
- [ ] Kontext ist vorhanden
- [ ] Es ist klar, ob daraus ein Issue werden sollte
- [ ] Antworten helfen bei Entscheidung, Verständnis oder Wissenstransfer
